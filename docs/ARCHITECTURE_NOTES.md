**Reading this:** `PlayerInventory`, `ItemData`, and `PrisonEventType` are the three real architectural pinch points of the gameplay code (the HUD and namespace entries are expected fan-out, not design smells). Any refactor of these three should be done behind a PR of its own with full test coverage run first, since `graphify affected` shows dozens of call sites for each.

## 3. Escape mechanics: no shared abstraction yet

```
graphify path "VentCover" "FakeBedDummy"
  VentCover --inherits--> MonoBehaviour <--inherits-- FakeBedDummy
```

`VentCover`, `FakeBedDummy`, and `PillowStash` are siblings only through `MonoBehaviour` — there is **no `IEscapeRoute` / `EscapeManager` abstraction** tying them together yet. This corroborates `Game_Features_And_Test_Coverage.md` §1.1: the escape *ingredients* exist independently, but the escape *system* (objective state machine + win condition) does not. When that system is built (tracked as the top roadmap item), it should sit above these three as a common interface/registry rather than each mechanic reporting escape progress ad hoc.

## 4. Guard detection call graph

`graphify query "guard detection and security alert system"` shows a clean, cycle-free chain:

```
GuardShiftController --schedules--> GuardFSM --drives--> GuardDetection
                                          |
                                          +--> .VerifyCellBedPresence() (night check)
                                          +--> .FindNonCompliantPrisoner() / .IsInSight() (patrol detection)
GuardFSM --reacts to--> PrisonEventType (via .OnPrisonScheduleChanged() / .SyncNightVerificationToSchedule())
```

`MorningShakedownSweeper` and `PrisonSecurityAlerts` sit one level above this as consumers, not part of the detection loop itself — confirms the security layer is additive and doesn't reach back into `GuardFSM`/`GuardDetection` internals (good separation).

## 5. Blast radius of the two riskiest types

`graphify affected "PlayerInventory" --depth 2`: 60+ call sites across crafting, every interactable, inventory/hotbar/notebook UI, multiplayer `Player`/`WeaponController`, and the EditMode test suite. **Any signature change here requires running the full `CraftingInventoryLootTests` suite at minimum.**

`graphify affected "PrisonEventType"` (not shown above, but same shape): the enum is the dependency root for `PrisonEventRules`, `PrisonEventExtensions`, `PrisonRoutineLabels`, `GuardFSM`, HUDs, and `PrisonRulesAndLabelsTests`. Adding a new phase value means touching all of those — verify against `PrisonRulesAndLabelsTests` after any change.

## 6. Isolated / weakly-connected nodes

The raw report flags 393 "isolated" nodes, but a direct query of `graphify-out/graph.json` for nodes with degree ≤ 1 whose source is a real `Assets/Scripts/*.cs` gameplay file returns **zero matches** — every one of the 102 gameplay scripts is wired into the graph with real edges. The isolated count is entirely `Packages/packages-lock.json` manifest entries, markdown doc headings, and built-in primitive/Unity types (`bool`, `float`, `Transform`, …), not undocumented game code.

## 7. Suggested next steps surfaced by the graph

1. **Build the `EscapeManager`/`IEscapeRoute` abstraction** (see §3) — the missing keystone for the stated main goal.
2. **Treat `PlayerInventory`, `ItemData`, `PrisonEventType` as protected APIs** — any PR touching their public surface should run the full EditMode suite and call out the affected list in the PR description (use `graphify affected "<Type>"` to generate it).
3. Re-run `graphify update .` (already wired via the post-commit/post-checkout git hooks) and refresh this doc after large refactors — it's auto-rebuilt locally with no API cost.
